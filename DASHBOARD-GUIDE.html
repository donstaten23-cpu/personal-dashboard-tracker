3. How data is stored (Supabase)

All data lives in a single Supabase table called dashboard_data. Each "section" of the dashboard is one row, identified by a tab_key string. The actual content is stored as a JSON object in the data column (type JSONB).

Table schema

dashboard_data
  user_id     uuid        (ties data to the signed-in user)
  tab_key     text        (identifies which section this row is for)
  data        jsonb       (the section's data as a JSON object)
  updated_at  timestamptz (set on every save)

Tab keys used by this app

tab_keyWhat's stored
dashboard-travel-dataAll trips: array of trip objects, each with items (flights, stay, activities…).
dashboard-budget-dataMonthly expenses, savings goal, pay period config.
dashboard-routines-dataRoutines array with name, frequency, completion history, streaks.
dashboard-workouts-dataWorkout sessions, exercises, sets/reps/weight history.
dashboard-education-dataDegree program, courses, grades.
dashboard-certs-dataCertifications list with dates and status.
dashboard-jobs-dataJob applications with status, company, notes.
portfolio-contentAll the text fields on the Work Portfolio page (editable in-place).
portfolio-img-{key}One row per portfolio image. data.src holds a base64 data URL. Keys: port-img-1 through port-img-5, art-dp-1 through art-dp-6, art-sp-1 through art-sp-6, etc.
user-settingsTheme colors, sidebar visibility, default landing page, greeting lines, content zoom.

How a section loads and saves data

// LOAD — call this inside any render function
var data = await loadTabData("dashboard-travel-data", { trips: [] });
// Returns saved data, or the default object if no row exists yet.

// SAVE — call this whenever the user makes a change
await saveTabData("dashboard-travel-data", data);
// Upserts the row (insert if new, update if existing).

4. Page architecture

How pages work

Every page is a <div class="page" id="page-{name}"> inside .main-content. Only one page has the active class at a time, which makes it visible. All others are hidden via CSS (display:none).

<!-- In the HTML -->
<div class="page active" id="page-dashboard">...</div>
<div class="page" id="page-travel">...</div>
<div class="page" id="page-budget">...</div>

How navigation works

Every sidebar button has a data-page="name" attribute. Clicking it calls switchPage("name"), which:

1
Removes active from all .page divs and adds it to #page-{name}
2
Removes active from all sidebar buttons and adds it to the clicked one
3
Runs any page-specific render logic (e.g. renderTravelTab(), renderSpendTab())

Lazy loading

Heavy pages like Workouts are rendered once on first visit using a _loaded flag on the root element. Pages like Travel and Budget re-render on every visit (since their data changes frequently). Routines renders once on sign-in and stays in the DOM.

6. How to add a new page

Four places to edit in index.html:

1
Add a sidebar button — inside <nav class="sidebar">, add:
<button class="sidebar-item" data-page="mypage">
  <span class="s-icon">◆</span> My Page
</button>
2
Add the page div — alongside the other .page divs in .main-content:
<div class="page" id="page-mypage">
  <div class="page-inner">
    <div id="mypageRoot"></div>
  </div>
</div>
3
Handle it in switchPage() — add a block inside the function:
if (page === "mypage") {
  renderMyPage();
}
4
Write the render function — add it anywhere in the script:
async function renderMyPage() {
  var root = document.getElementById('mypageRoot');
  var data = await loadTabData('my-page-data', { items: [] });
  root.innerHTML = '...';
  // set up event listeners here
  root.addEventListener('click', function(e) {
    // handle actions via e.target.closest('[data-action]')
  });
}
5
(Optional) Add it to settings visibility toggles — find the sidebarPages array inside renderSettingsPage() and add { id: 'mypage', label: 'My Page' }.

8. Google Calendar integration

The calendar uses Google's OAuth 2.0 implicit grant flow via the Google Identity Services (GIS) library. No server-side code needed — the browser gets a short-lived access token directly.

Flow

1
User clicks Connect Google Calendar → connectGoogleCalendar() calls google.accounts.oauth2.initTokenClient(...).requestAccessToken()
2
Google shows its consent screen. On approval it returns an access token (valid ~1 hour).
3
Token is saved in sessionStorage under key "gcal-token" with its expiry. This survives a page refresh but not closing the tab.
4
fetchAndRenderCalendarEvents() hits the Google Calendar REST API and renders events grouped by day.
5
If the API returns 401 (expired), the token is cleared and the card reverts to the Connect button.

9. Settings system

Settings are stored as a single JSON object in Supabase under tab_key = "user-settings". They're loaded on sign-in by loadSettings() and applied by applySettings().

What gets saved

FieldControls
accentThe main accent color (--accent variable)
bgColor, cardColor, textColor, sidebarColor, borderColorAll other theme colors
cardStyle"rounded" (default), "flat", or "glass"
contentZoomScale factor for the main content area
siteTitleThe name shown at the top of the sidebar
logoDataUrlBase64 logo image for the sidebar header
userName, partnerNameDisplay names used in greetings and budget labels
greetingLinesCustom greeting lines (one per line in the textarea); random one is picked each time
hiddenPagesArray of page IDs hidden from the sidebar
defaultPageWhich page to land on after sign-in (defaults to "dashboard")
pageLabelsCustom eyebrow labels per section (e.g. "TRAVEL TRACKER")

How settings are applied
